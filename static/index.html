1. Ingestion & Parsing

  • Format-specific parsers (PDF, DOCX, HTML, Markdown, Text)
  • Structure preservation (headings, page numbers)
  • Text cleaning: normalize whitespace, remove artifacts
  • Metadata extraction: filename, timestamps, file type

2. Chunking Strategy

  • Semantic chunking by headings/sections first
  • Sentence-boundary splitting for large sections
  • ~300 tokens per chunk with ~50 token overlap
  • Minimum chunk size filter (50 tokens)
  • Each chunk tagged with source metadata

3. Embedding & Indexing

  • all-MiniLM-L6-v2 (384-dim, normalized)
  • ChromaDB with HNSW cosine similarity index
  • BM25 index for keyword/lexical matching
  • Incremental updates via upsert

4. Retrieval & Reranking

  • Query rewriting: abbreviation expansion, cleanup
  • Hybrid search: vector + BM25 with RRF fusion
  • Configurable alpha (default 0.7 vector weight)
  • Re-ranking via embedding similarity rescoring
  • Top-k filtering (20 candidates -> 5 final)

5. Generation & Grounding

  • Ollama LLM (llama3.2) with low temperature (0.1)
  • System prompt enforces source-only answers
  • Mandatory citation format [Source: filename]
  • Abstention when context is insufficient
  • Context deduplication and compression

6. Safety & Guardrails

  • Prompt injection detection (pattern matching)
  • Query length limits
  • Retrieval confidence thresholds
  • Graceful fallback: "I don't know" responses
  • Abstention when retrieval score too low

Evaluation Framework

A. Chunking / Ingestion Evaluation

Goal: Chunks should be coherent and retrievable.

  • Manual audits on sampled set: coherence, duplication rate, metadata correctness
  • Automated checks: avg chunk length, overlap %, parser error rates
  • Regression tests on known "hard" documents (tables, PDFs)
  • Monitor: chunk size distribution, parser failures per format

B. Retrieval Evaluation

Goal: Relevant chunks appear in top-k results.

  • Labeled dataset: (query, relevant chunks/docs)
  • Recall@K: fraction of queries with relevant chunk in top-k
  • MRR / nDCG: reward correct ranking order
  • Slice by doc type, query type, freshness
  • Weak supervision: synthetic Q/A pairs from docs

C. Reranker Evaluation

Goal: Reranking improves precision.

  • Compare nDCG@10 before vs after reranking
  • Latency/cost tradeoff analysis
  • Error analysis: bias toward longer passages
  • A/B test: with/without reranking

D. Generation Evaluation

Goal: Answers are correct, grounded, and cited.

  • Correctness: Does it answer the question?
  • Faithfulness: Is it supported by retrieved context?
  • Citation quality: Do citations back the claims?
  • Human grading rubric + LLM-as-judge
  • Attribution check: each sentence maps to a cited chunk

E. End-to-End Evaluation

  • Task success rate, user satisfaction
  • Latency p50/p95, cost per query
  • Abstention rate balance (too high = useless, too low = hallucinations)
  • User feedback loop (thumbs up/down)
  • Failure case collection for eval set growth

F. Continuous Improvement

  • Log all queries, retrieved docs, outputs, latency
  • Track drift: embedding distribution, score distributions
  • Collect failure cases into regression test suite
  • Tune: chunking params, k, reranker thresholds
  • Add domain-specific synonym expansions
